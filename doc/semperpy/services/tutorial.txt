Services - client-server communication
**************************************

    We are going to see a simple example of how to implement a server and a client talking to each other. We
    will then move on a more elaborated application. Since these examples are more elaborated, we will try and
    make use of more and more tools available in SemperPy, we will also show how the :mod:'Language' module
    which is useful in that case.

A calculator service
---------------------

.. _services_calculator:

    We implement a server which offer a calculator service: it is able to compute +,- and to return 
    the result. The request is expected in the following format::

    	{
    		operator = '+',
    		arguments = [2,3,4]
    	}

    For the sum any number of arguments is handled, for the substraction, only two parameters are accepted. 
    The result is returned to the client as::

    	9

    This is a bit more output than we would like, but it is inserted by *twisted*.

    By default, the server we creates uses a mode in which it does not block but executes the code
    for the service in a different thread. The client is kept waiting for a response with the 
    connection opened for as long as it takes. The server goes back to accepting connections
    immediately.

    It is quite trivial to write a client-server interaction using these tools. The difficulty
    lies in fact in making sure that the data which is communicated between the client and 
    the server is correct. Instead of checking on the server side that everything which is
    received is correct, it would be better to check all this at client level:

    * so that no invalid requests are sent through the network, wasting bandwith and resources
    * so that the server does not fail after a long processing of data because of it is invalid,
    * so that the server which is supposed to do complicated things can be simplified by relying on validated data

    The :mod:`Language` module should enable us to do this in a parameterised way.

Serving services and pages
--------------------------

Services
^^^^^^^^

    You can register as many services as you wish, the limit being how much memory is required
    and how much the hardware can take. Services can be installed in a hierarchy, this can 
    sometimes be useful to present meaningful urls to the user:

    	Server.registerService("retrieve",retrieve_func)
    	Server.registerService(["retrieve","help"],help_func)
    	Server.registerService(["retrieve","status"],status_func)

    The family of services are grouped under the main name *retrieve* and can be accessed as follows:

    	* host:port:/retrieve
    	* host:post:/retrieve/help
    	* host:post/retrieve/status

    This grouping is offered just to offer a family of urls to the user if it makes the services easier to
    understand, otherwise the different services can be regsitered separately.

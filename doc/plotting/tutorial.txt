Observation statistic - plotting
********************************

    The plotting system can be alimented by different sources of data or **storage** systems, for example, an SQL database or a text file. 
    It is flexible enough to enable:

        * a high level of customization of the plots by the end-user
        * an easy addition of new plot types to find better representations of features in the data or to handle new data types

    Those two features have a cost: **complexity**. Some sort of compromise between the complexity
    of the software and the flexibility of the system had to be made. The plotting
    system presents capabilities for:
        
        * retrieving data from a database, a file or any other medium for which a **reader** can be provided
        * computing statistics from intermediate results which were generated during a **computing** stage.

    The plotting systems handles:

        * curves
        * bars, histograms
        * dots
        * scatter plots
        * plot titles
        * legends
        * axis titles
        * color maps

    This is carried out automatically, most of the features can be customized.

The doc-plot-curve relationship
-------------------------------

    We have three main classes involved in creating this plot:

        * obsdocument
        * timeseriesplot
        * curve

    Let's first explain how their relationship influences the production of plots
    (let's call it the doc-plot-curve relationship). This relationship conditions all
    kinds of plots.
    In the observing system, there is a list of keywords (dimensions) which 
    determine the choice of observations (this list could possibly evolve in the future):

        * kx
        * kt
        * date
        * level
        * type of level
        * domain_name
        * variable (original name of the ODS variable)
        * statistic name
        * usage (used, all, passive)

    We can add, in order to make sure that each observation is unique (this list could possibly evolve in the future):

        * expver or expid, the experiment id
        * source, this is an arbitrary parameter which is necessary for the system to function properly, at the time of writing, it could be:
            - **oper** generated by a system handled by OPS
            - **exp** generated by a research experiment
            - **odas** generated by the ocean analysis system
            - **aer** generated by the aerosol system.

    All these properties can be defined at any level in the doc-plot-curve relationship, but 
    the lower down they are defined (towards the curve)  the more precedence they have.

            variable = 'xvec',
            domain = ['global','n.hem'],

            level = 1000,
            statistic = 'impact_per_anl',
            layout = [1,2],              
            has_title = True,

Observation description file
----------------------------

    In the examples below, *kt* and *kx* are specified at curve level. Depending of the
    type of plot we want to generate, the distribution of *kt* / *kx* combinations can change.
    In an attempt to avoid comlicating the system too much we settled for the following
    approach:
    
        * at document level, you can optionally specify a file name containing a list of kt and kx. A plot will be generated for each *kt* / *kx* combination.
        * at plot level, you can also specifiy a filename, a curve (or bar) will be generated for each *kt* / *kx* combination.
        * at curve level, you can specify *kt* / *kx* combinations manually, no file can be specified.

Observation file
^^^^^^^^^^^^^^^^

    The file timeseries_all.def contains a list of groups of observations to be plotted :download:`kt/kx combinations`<timeseries_all.def>`.

Date file
^^^^^^^^^^

    Whenever you need to plot long timeseries or aggregate statistics over a long period of time,
    the data you need to retrieve from the database might be tagged with different experiment IDs.
    The curve object accepts a keyword: **datefile** which is a file which defines date intervals
    together with experiment IDS (expver):

        .. literalinclude:: dates.def

    :note: the name of the section (between square brackets), are abitrary, anything can be used provded that it is unique.

Output name
^^^^^^^^^^^

    The **output** keyword in this directive defines the filename of the graphics file(s) to generate.
    When many plots are generated, it is preferable to follow some file naming convention. The file name
    can be parameterized in a way which is rather similar to the titles (although not everything which
    works for a title might work for the file name)::

        output = 'timeseries_<name>_<domain_name>_<statistic>.png',

    If the filename is not parameterized in satisfactory way, the program will complain and stop in order
    to avoid overwriting an existing file. In doubt, you can always add <count> in the filename, which is 
    a counter per unique file name, this ensures that file names are unique.

Plotting directly from ODS files
--------------------------------

    A mechanism to read data from ODS files has been put in place to enable the creation of plots
    handling raw data. Some standard plots are available, while new ones can quite easily be programmed
    requiring a little knowledge of **matplotlib**. Since observations are accessed directly from an ODS file,
    ODS plot directives are quite similar to Obsstat directives.

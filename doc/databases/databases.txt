.. _databases:

Databases
*********

    This chapter is intended both for end-users and administrators / developers.

    The majority of the applications in Semperpy work on the same model whenever possible:

        - raw data is extracted from files
        - partial or final statistics are computed for different types of data
        - those results are stored in a database,
        - later on, partial or final statistics are read from the database, aggregated over different dimensions if applicable and printed or plotted etc...

    The idea of that model is to extract information from large data files and store much smaller information resulting
    usually from a geographical aggregation of data and computing statistics.

    The type of database used is an "SQL database", the particular implementation installed is PostGreSQL. It works following a client/server model:

        - the server runs on a machine (in our case the Data Portal)
        - clients run on different machines (in our case particularly on NCCS nodes)

    Reading and writing from and to a database is done through the network. There are sometimes restrictions for security
    reasons as of which nodes can access a database server. On NCCS, please note the PostGreSQL server can only be
    accessed by interactive nodes (not computing nodes). This could cause problems but there are workarounds used.
    example, in operations.


Physical databases, logical databases and security
--------------------------------------------------

    Like in all systems, there is security involved in database reading and writing. PostGreSQL enables to us to define
    physical databases, these are created manually by database administrators. The database server keeps the definition
    of users with different credentials, just as well as Linux. Credentials are given to a particular user on a particular
    database. In order to avoid having to make too many requests from the administrators and to keep good security
    practice, the following scheme was put in place:

        - logical databases are created in physical databases in order to keep the freedom of creating and deleting databases,
        - three levels of user credentials are used,
        - only a one-off setup is necessary by the database administrators

    Three different profiles of users were identified:

        - users who need to read data from any database
        - users who run research experiments and who need to store statistics for those experiments
        - users who run operational jobs and need to store statistics

    In order to achive the above, the following strategy was implemented:

        - two physical databases were created by the administrators (gmao_stats and semper)
        - three database users were created by the administrators (gmao_user, gmao_rd, gmao_ops)

    The following credentials were setup:

        +-----------------------+-----------------------+-----------------------+
        |User / Database        |gmao_stats             |semperpy               |
        +=======================+=======================+=======================+
        |gmao_user              |read                   |read                   |
        +-----------------------+-----------------------+-----------------------+
        |gmao_rd                |read                   |read/write             |
        +-----------------------+-----------------------+-----------------------+
        |gmao_ops               |read/write             |read                   |
        +-----------------------+-----------------------+-----------------------+

    Any user of SemperPy is a **gmao_user**, therefore any user can read any database, this makes sense, it would be rather
    impractical to need to setup a password just to plot some dats. By default, for user **gmao_user**, nothing needs
    to be done, invoking SemperPy to generate plots works from any database.

.. _pgpass:

    If you run research experiments and would like to store statistics, you will need to request a password from the
    **SemperPy Administrator**, no one has been officially been given that function yet, but requesting the password
    from Austin or Denis would be a good start. PostGreSQL requires that the password of the database user be
    stored in the user's home directory in a file called .pgpass which cannot be read by anyone else::

        cd $HOME
        touch .pgpass
        chmod go-r .pgpass

    You will then need to write this line in the file::

        dpdb:*:*:gmao_rd:password-given-to-you

    You can then write and delete data to and from the **semper** database.

    If you need to write data into an operational database, then you will need to request the password and
    follow the instructions above.

    You will then need to write this line in the file::

        dpdb:*:*:gmao_ops:password-given-to-you

    You can then write and delete data to and from the **gmao_stats** database.

    The security put in place through two different physical databases and three different user IDs is only
    for safety purposes in order to avoid deleting or overwriting data.

    During the lifetime of SemperPy and the deployment of new applications, new databases will be created and probalby deleted,
    so in order to keep the freedom of doing this at will, logical databases have been created.
    As of now, **logical databases** are the only databases we will talk about. A **logical database** is a storage 
    compartment with a special name (for the expert, a PostGreSQL schema) to enable a different storage place for data.
    The advantage of **logical databases** is that many of these can be created either by the SemperPy administrator 
    or the end user. This enables to create logical chunks in the data (e.g. observations, observation impacts, 
    forecast skill scores, operational, experimental, test, etc...). Those chunks make it then practical to  
    delete data, copy data, duplicate data, dump data etc...


.. _databasedefinition:

Database definition
-------------------
    
    In the configuration directory of SemperPy, there is a directory called db, containing a configuration file
    called databases.def. This is the system configuration file defining the database used system-wide both for
    operations, research experiments and tests. Let's examine the definition of two similar databases, im_ops and 
    im_exp, the former is the database for operational impact statistics, the latter is the database for 
    experimental impact statistics::


        [im_ops]
        host = dpdb
        database = gmao_stats
        schema = schema/ob_schema.def
        recordtype = obsstat
        userw = gmao_ops
        userr = gmao_user
        password=hemperUI9#
        grantr = gmao_user
        engine = psql
        buffer = 2000
        protected = yes
        comment = operational database for observation impact statistics

        [im_exp]
        host = dpdb
        database = semper
        schema = schema/ob_schema.def
        recordtype = obsstat
        userw = gmao_rd
        userr = gmao_user
        password=hemperUI9#
        grantr = gmao_user
        engine = psql
        buffer = 2000
        protected = yes
        comment = experiment database for observation impact statistics
        
    The keywords are defined as follows:

        - [im_ops] between square brackets is the name of the **logical database**.
        - host, the host running the PostGreSQL server, in our case **dbdb**.
        - database, the **physical database** holding the **logical database**, gmao_stats for operations and semper for research experiments and private databases.
        - schema, describes which file holds the definition of the database keywords for SemperPy to be able to process correctly. There are currently three different schemas: schema/ob_schema for observation statistics (including impact atatistics), schema/sc_schema for forecast skill scores, and schema/pr_schema for statistic on limited areas of analysis gridded data (fields).
        - recordtype, describes which directive defines an pre-initialized record to work with that particular database and schema, obsstat for observation statistics, scores for forecast skill scores, and product for statistics of analyses.
        - userw, defines which user is allowed to write in that database.
        - userr is always gmao_user (the user who can read all databases)
        - password is gmao_user's password in PostGreSQL, this is public because that user can only read data.
        - grantr is always gmao_user (because that user can read all databases)
        - engine is always psql (for PostGreSQL), but could possibly be mysql or other.
        - buffer, defines the number of records to store int the database to be held in memory before flushing to the PostGreSQL server. The bigger that number, the less communication with the server and the smaller SQL requests for writing data, but the bigger that number the more memory is needed for a computing job. This hasn't been tuned much.
        - the protected flag indicates that this database is important and it can therefore not be deleted by a standard command, an administrator is required to do this. This is to prevent deleting an big database with a single command by mistake.
        - comment, is the line which appears in the dynamic documentation when issuing the command::
            
            spy_doc databases

        The command **spy_doc databases** reads this configuration file and displays:

            - the name of the logical database,
            - the type of database (recordtype)
            - the writer, which PostGreSQL user is allowed to write in that database (userw),
            - the comment

        The command **spy_doc database <database>** display the contents of the configuration together with
        the list of keywords associated with that database (recordtype)::

            spy_doc databases im_ops
 
            im_ops:
            buffer     = 2000
            comment    = operational database for observation impact statistics
            database   = gmao_stats
            engine     = psql
            grantr     = gmao_user
            host       = dpdb
            password   = hemperUI9#
            protected  = yes
            recordtype = obsstat
            schema     = schema/ob_schema.def
            userr      = gmao_user
            userw      = gmao_ops
             
            Keywords for im_ops: count, date, domain_name, east, expver, kt, kx, level, levtype, north, source, south, statistic, type, usage, value, variable, west
             
             
            generated on 2012-05-02 23:57:37
 
Scripts for database management
-------------------------------

    There are a few scripts to help both the administrator and the end user
    to handle SemperPy databases:

        - **spy_createdb**: creates (and possibly overwrites) a database
        - **spy_dropdb**: deletes a whole logical database
        - **spy_deletedb**: deletes selected data from a database (e.g. experiment)
        - **spy_db2txt**: dumps whole or part of the contents of a database into a text file
        - **spy_txt2db**: loads the contents of the text file generated by spy_db2txt or by computing scores into a text file into a database
        - **spy_sql**: starts the PostGreSQL shell on the right physical database. 

.. _createdb:

Creating a database
^^^^^^^^^^^^^^^^^^^

    You can use the script called **spy_createdb** to create a database. The database needs to be part of
    the database definition so that it is known by the system. Here is the syntax of the command::

        spy_createdb --help
        usage: createdb.py [-h] [--drop] [--drop_only] [--grant_only] database

        generates the SQL instructions to create a logical database

        positional arguments:
          database      name of the database to be created

        optional arguments:
          -h, --help    show this help message and exit
          --drop        drop the database if it already exists
          --drop_only   drop the database if it already exists and does nothing else
          --grant_only  only generate the sql GRANT instructions

    The different types of databases (to date) are:
    
        - obsstat, an observation and observation impact statistic database
        - score, an forecast skill score database
        - product, a product based database, statistics over limited domains of analyses

    The option --drop_only is equivalent to the command **spy_dropdb**. The --drop options instructs to
    first delete the existing database. The option --grant_only is irrelevant. Databases which are protected 
    (protected = yes) are not dropped by this command, even if the --drop or --drop_only flags are used.

    Example::

        spy_createdb im_ops
        create schema im_ops; 

        create table im_ops.domain (
        id serial primary key,
        north float,
        west float,
        south float,
        east float,
        domain_name varchar(25),
        unique (north,west,south,east,domain_name)
        );


                
        create table im_ops.stats (
        id serial primary key,
        level float,
        levtype varchar(5),
        expver varchar(20),
        source varchar(10),
        variable varchar(15),
        statistic varchar(15),
        type varchar(10),
        kx int,
        kt int,
        usage varchar(10),
        domain_id int not null references im_ops.domain(id),
        unique (level,expver,source,variable,statistic,type,kx,kt,usage,domain_id)
        );
        create index stats_0_idx on im_ops.stats (type);
        create index stats_1_idx on im_ops.stats (expver);
        create index stats_2_idx on im_ops.stats (variable);
        create index stats_3_idx on im_ops.stats (statistic);
        create index stats_4_idx on im_ops.stats (kx);
        create index stats_5_idx on im_ops.stats (kt);
        create index stats_6_idx on im_ops.stats (usage);
        create index stats_7_idx on im_ops.stats (level);
        create index stats_8_idx on im_ops.stats (usage);
        create index stats_9_idx on im_ops.stats (variable,level,type,domain_id,expver,kx,kt,usage);

                
        create table im_ops.value (
        date int not null,
        value float,
        count float,
        stats_id int not null references im_ops.stats (id)
        );
        create index value_10_idx on im_ops.value (date);
        create index value_11_idx on im_ops.value (stats_id);
        alter table im_ops.value add primary key (date,stats_id);

                

        create or replace view im_ops.v_view (north,west,south,east,domain_name,level,levtype,expver,source,variable,statistic,type,kx,kt,usage,date,value,count)
        as select north,west,south,east,domain_name,level,levtype,expver,source,variable,statistic,type,kx,kt,usage,date,value,count
        from im_ops.domain,im_ops.stats,im_ops.value where stats.domain_id = domain.id and value.stats_id = stats.id;

        grant usage on schema im_ops to gmao_user;
        grant select on table im_ops.domain to gmao_user;
        grant select on table im_ops.stats to gmao_user;
        grant select on table im_ops.value to gmao_user;
        grant select on table im_ops.v_view to gmao_user;

    The command issues the SQL commands into the standard output. In order to have these commands executed,
    the output needs to be sent to the PostGreSQL shell, for example::

        spy_createdb im_ops | spy_sql -w im_ops

    Please check the paragraph about :ref:`postgresql`. In order to create a database, you are required to
    have the credentials to write into a physical database (semper for gmao_rd and gmao_stats for gmao_ops).
    Those credentials can be obtained by getting the password for the right user and creating or updating
    your :ref:`.pgpass <pgpass>` file.

    You can create your own database, so that you can experiment and perhaps just drop it later on. To do so,
    you need to:
    
        - define your database
        - let SemperPy know about your definition
        - use the spy_createdb script
        - specify that database name in your computing and plotting scripts.

    To define your database, please refer to :ref:`configuration` for background information. In your private
    configuration directory, you need to create a directory called **db**, in which you create a file called
    databases.def. That file can contain database definition(s)::

        [my_db]
        host = dpdb
        database = semper            # or gmao_stats depending on your credentials
        schema = schema/ob_schema.def
        recordtype = obsstat
        userw = gmao_rd              # or gmao_ops if you specified database = gmao_stats
        userr = gmao_user
        password=hemperUI9#
        grantr = gmao_user
        engine = psql
        buffer = 2000
        protected = no               # unless you are a little paranoid
        comment = my own database I can experiment with

    You then need to let SemperPy know that you have your own configuration directory::

        setenv SEMPERPY_CONFIG /my/path/to/my/configuration/directory:$SEMPERPY_CONFIG
        or
        export SEMPERPY_CONFIG=/my/path/to/my/configuration/directory:$SEMPERPY_CONFIG

    From that stage, you database is part of the system. You can create it and start using it. If
    you type::

        spy_doc databases
        fc_exp  : type = score    writer = gmao_rd  experiment database for forecast scores
        fc_ops  : type = score    writer = gmao_ops operational database for forecast scores
        im_exp  : type = obsstat  writer = gmao_rd  experiment database for observation impact statistics
        im_ops  : type = obsstat  writer = gmao_ops operational database for observation impact statistics
        my_db   : type = obsstat  writer = gmao_rd my own database I can experiment with <---------
        ob_exp  : type = obsstat  writer = gmao_rd  experiment database for observation statistics
        ob_ops  : type = obsstat  writer = gmao_ops operational database for observation statistics
        ocean   : type = obsstat  writer = gmao_rd  database for observation statistics for the ocean
        pr_ops  : type = product  writer = gmao_ops test database for statistics on field products
        test_exp: type = obsstat  writer = gmao_rd  test database for experiments, can be deleted at anytime
        test_ops: type = obsstat  writer = gmao_ops test database for operations, can be deleted at anytime

        generated on 2012-05-03 23:21:28

    and obtain information::

        spy_doc databases my_db

        my_db:
        buffer     = 2000
        comment    = my own database I can experiment with
        database   = semper
        engine     = psql
        grantr     = gmao_user
        host       = dpdb
        password   = hemperUI9#
        protected  = no
        recordtype = obsstat
        schema     = schema/ob_schema.def
        userr      = gmao_user
        userw      = gmao_rd
         
        Keywords for my_db: count, date, domain_name, east, expver, kt, kx, level, levtype, north, source, south, statistic, type, usage, value, variable, west
         
         
        generated on 2012-05-03 23:24:12


Dropping a whole database
^^^^^^^^^^^^^^^^^^^^^^^^^

    This command should be used with great caution provided that a whole database can be deleted using this
    command in less than a second independently of its size. This command is provided to delete users' own
    databases when they are not used anymore, *official* databases are protected and will not be deleted by
    this command. This command (which actually invokes **spy_createdb** with the right arguments), writes the SQL
    commands to delete the database in the standard output.  In order to have these commands executed, 
    the output needs to be sent to the PostGreSQL shell, for example::

        spy_dropdb my_db | spy_sql -w my_db

Deleting part of a database
^^^^^^^^^^^^^^^^^^^^^^^^^^^
   
    Most of the time when computing statistics, you are given the option to overwrite the data, so that if
    you need to re-compute statistics, you can just replace the contents of a database for one particular
    experiment and dates etc.. It could sometimes be useful to be able to delete a whole expriment because
    it is not useful anymore and as a good person, you delete it to free some space, or just because 
    you want to re-compute all the statistics and it is much faster to *write* than to *overwrite*::

        py_deletedb --help
        usage: deletedb.py [-h] [--begindate BEGINDATE] [--enddate ENDDATE]
                           [--expver EXPVER] [--level LEVEL] [--kx KX] [--kt KT]
                           [--domain_name DOMAIN_NAME] [--statistic STATISTIC]
                           [--usage {used,passive,rejected}] [--variable VARIABLE]
                           [--dryrun]
                           database

        Delete selected statistics. Please not that not all keywords are valid for all
        databases, support is provided, if a wrong keyword is specified

        positional arguments:
          database              name of the database to delete data from

        optional arguments:
          -h, --help            show this help message and exit
          --begindate BEGINDATE
                                start date in the form YYYYMMDDHH
          --enddate ENDDATE     end date in the form YYYYMMDDHH
          --expver EXPVER       experiment ID of the data to dump
          --level LEVEL         list of pressure level to dump
          --kx KX               list of kx
          --kt KT               list of kt
          --domain_name DOMAIN_NAME
                                name of a geographical area
          --statistic STATISTIC
                                name of a statistic
          --usage {used,passive,rejected}
                                data usage flag
          --variable VARIABLE   name of a variable
          --dryrun              only displays sql commands without executing them

    Please be careful with deleting data, particularly because not specifying a keyword is equivalent to
    "delete all of these". This command will not delete anything if nothing is specified. Please note the
    argument *--dryrun* which displays the SQL command for deletion but does not execute them. This could
    give you an idea of the number of records to delete and might help you correct mistakes in your selection.

    If the database you want to delete from is protected, you will need to confirm your intention to delete
    from that database.

    :note: unlike dropping a database, deleting selected data can take a very long time, if you try to delete a very long experiment, it can literally take hours.


Dumping and loading a database
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

    Just as well as you can select data to be deleted from a database, you can select data to be dumped into
    a text file. The script **spy_db2txt** has arguments which are very similar to **spy_deletedb**::

        spy_db2txt --help
        usage: db2txt.py [-h] [--begindate BEGINDATE] [--enddate ENDDATE]
                         [--expver EXPVER] [--level LEVEL] [--kx KX] [--kt KT]
                         [--domain_name DOMAIN_NAME] [--statistic STATISTIC]
                         [--usage {used,passive,rejected}] [--variable VARIABLE]
                         [--append]
                         database filename

        Dump the whole contents or part of an SQL database into a text file. Please
        not that not all keywords are valid for all databases, support is provided, if
        a wrong keyword is specified

        positional arguments:
          database              name of the database to dump
          filename              filename of the text file to create

        optional arguments:
          -h, --help            show this help message and exit
          --begindate BEGINDATE
                                start date in the form YYYYMMDDHH
          --enddate ENDDATE     end date in the form YYYYMMDDHH
          --expver EXPVER       experiment ID of the data to dump
          --level LEVEL         list of pressure level to dump
          --kx KX               list of kx
          --kt KT               list of kt
          --domain_name DOMAIN_NAME
                                name of a geographical area
          --statistic STATISTIC
                                name of a (raw) statistic
          --usage {used,passive,rejected}
                                data usage flag
          --variable VARIABLE   name of a variable
          --append              if the file exists, append data to the file

    For example, if you want to save all observation impact statistics for radiosondes in March 2012::

        spy_db2txt --expver=e572p2_fp --kx=120,220 --kt=4,5,11,33,44 --begindate=2012030100 --enddate=2012033100 im_ops  rs_Mar2012.txt

    We speficied the experiment id (expver) to make sure that if there were any parallel runs at the same time, these would not be
    dumped with the rest of the data.

    The file rs_Mar2012.txt is generated, the first few lines look like this::

        count|date|domain_name|east|expver|kt|kx|level|levtype|north|source|south|statistic|type|usage|value|variable|west
        2916.0|2012032200|global|180.0|e572p2_fp|11|120|700.0|pl|90.0|oper|-90.0|beneficial|im|used|1450.0|xvec|-180.0
        2677.0|2012031900|global|180.0|e572p2_fp|11|120|700.0|pl|90.0|oper|-90.0|beneficial|im|used|1342.0|xvec|-180.0
        2634.0|2012031800|global|180.0|e572p2_fp|11|120|700.0|pl|90.0|oper|-90.0|beneficial|im|used|1260.0|xvec|-180.0
        2713.0|2012031600|global|180.0|e572p2_fp|11|120|700.0|pl|90.0|oper|-90.0|beneficial|im|used|1368.0|xvec|-180.0
        2589.0|2012031400|global|180.0|e572p2_fp|11|120|700.0|pl|90.0|oper|-90.0|beneficial|im|used|1401.0|xvec|-180.0
        2721.0|2012031700|global|180.0|e572p2_fp|11|120|700.0|pl|90.0|oper|-90.0|beneficial|im|used|1467.0|xvec|-180.0

    This way you can dump a whole database, a whole experiment or some more selective data. When computing statistics from 
    original files, you can set the option to store the statistics in a text file instead of a database. In that case, the file
    created has exactly the same format as the files created by a database dump (please see, for example :ref:`obsstattutorial`).
    You can then use this text file to load the data into another database (duplication of database or 
    "export" onto a different site). In order to load the data you need to invoke the command **spy_txt2db**::

        spy_txt2db --help
        usage: txt2db.py [-h] [--overwrite] [--no_check_domains] [--dry_run]
                         [--missing_value MISSING_VALUE]
                         database filename [filename ...]

        Populates an SQL database using data from a text file

        positional arguments:
          database              name of the database to populate
          filename              filename(s) of the text files containing data, more
                                than one filename can be specified

        optional arguments:
          -h, --help            show this help message and exit
          --overwrite           overwrites data in the database if already present
          --no_check_domains    don't check that domain coordinates defined in the
                                text file match those defined in SemperPy
          --dry_run             don't check that domain coordinates defined in the
                                text file match those defined in SemperPy
          --missing_value MISSING_VALUE
                                value of missing value if applicable, lines containing
                                missing value are not loaded into the database


    To reload the data we just dumped, we can use the command::

        spy_txt2db test_exp rs_Mar2012.txt

    The data is then loaded. The loading program checks that the columns in the text files corresponds to the keywords of the 
    database it needs to be loaded into, if these don't match the program fails.

    :note: some databases are quite small, for example, the contents of the observation impact statistic database from 16 August 2010 until 2 May 2012 is a 390MB text file which gzips to 44MB. This is how I can easily have copies of operational databases on my laptop. For observation statistics, it is a different story, one month of data (Jan 2012) is 786MB.

.. _postgresql:

Using the PostGreSQL shell
^^^^^^^^^^^^^^^^^^^^^^^^^^

        The PostGreSQL shell is the binary utility delivered with the database package which enbles users to interact 
        with databases using the SQL language. Although the end user should not have to worry much about using it,
        it is useful to know how to start it to run the **spy_createdb** or the **spy_dropdb** commands. It can also be 
        useful to explore a database and find out what is in it. It sometimes disconcerting when you compute statistics
        and when you plot them you end up with empty plots. Using the database shell to double check if the data is in 
        the database could be useful.

        There are three different contexts in which you might need to start the PostGreSQL shell:

            - you just want to browse the database, in this case you will need to have the gmao_user password in your *.pgpass* file, please see :ref:`.pgpass <pgpass>` about how to create that file. You will need to add the line::
        
                dpdb:*:*:gmao_user:hemperUI9#

            - you want to have write access to or create a logical database in the physical database semper. In that case you *.pgpass* files needs to contain the line::

                dpdb:*:*:gmao_rd:<the password you were given>

            - you want to have write access to or create a logical database in the physical database gmao_stats. In that case you *.pgpass* files needs to contain the line::

                dpdb:*:*:gmao_ops:<the password you were given>

        Invoking the shell is easy::

            spy_sql <database>
            # read-only mode

            spy_sql -w <database>
            # write / creation mode, depending on where the logical database <database> is,
            # you will need the credentials of either gmao_rd or gmao_ops.

        Once you have started the shell, here are a few commands which might prove useful to explore a database. Let's use
        a real example so that you can try these out (operational database of observation impact statistics im_ops)::

            spy_sql im_ops
            gmao_stats=#

        Find out all experiments in that database::

            gmao_stats=# select distinct expver from im_ops.v_view;
              expver   
            -----------
             e571p1_fp
             e572p2_fp
             e571_fp
             e572p1_fp
             e562p3_fp
             e562p4_fp
             e562p5_fp
             e572_fp
